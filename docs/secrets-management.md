# Управление секретами и конфигурацией

В этом документе описаны основные проблемы хранения секретов и конфигурационных параметров, а также рассмотрены различные подходы к их безопасному менеджменту.

Проблема управления секретами и конфигурацией выходит далеко за рамки пользовательских dotfiles. Можно выделить три основных сценария (use cases), каждый из которых имеет свою специфику:

1. **Кодовые проекты (Dev Projects):** Основной и самый сложный кейс. Наличие нескольких окружений (Dev, Staging, Prod), командная разработка, полноценные конфигурации и фича-флаги. По своей сути конфигурация, секреты и фича-флаги — это одно и то же в плане использования приложением. Разница лишь в том, что секреты должны быть скрыты и зашифрованы, а обычные конфиги и флаги могут быть публичными.
2. **Self-hosted Home Lab:** Единый репозиторий, содержащий множество `docker-compose` файлов с различными сервисами и инфраструктурной конфигурацией.
3. **Dotfiles:** Конфигурация локального рабочего окружения пользователя (скрипты, алиасы, настройки утилит).

---

## Главные проблемы и задачи

### 1. Невозможность хранения секретов в Git в открытом виде

Это базовая проблема, о которой знают почти все. Секреты (API-ключи, пароли, токены, UUID приватных проектов) нельзя коммитить в публичные и даже приватные репозитории в формате plaintext из-за риска компрометации.

### 2. Почему хранить секреты в репозитории — плохая идея

Главная концептуальная причина отказа от хранения наборов секретов (даже в зашифрованном виде, например через `sops` или `dotenvx`) внутри репозитория — **нарушение принципов 12-factor app**.

Согласно этой методологии, секреты — это часть конфигурации, а **конфигурация должна жить строго отдельно от реализации (кода)**. Хранение секретов в Git приводит к архитектурным проблемам:

- **Смешивание жизненных циклов:** Изменение токена или пароля не должно порождать новый коммит в репозитории. Ротация секретов — это процесс изменения конфигурации деплоя, а не изменения логики приложения.
- **Утечка знаний об окружениях в код:** Если одну и ту же реализацию нужно использовать в разных местах с разными наборами токенов, в репозитории неизбежно появляется костыльная логика выбора набора (if-else, разные файлы конфигурации в репо).
- **Деплой vs Реализация:** Выбор токенов — это сугубо вопрос развертывания (деплоя), который должен разрешаться "снаружи". Инструмент должен получать нужные секреты из своего окружения, не зная, откуда они взялись.

### 3. Разнообразие способов доставки секретов в инструменты

Различные системы и утилиты по-разному умеют работать с секретами и конфигурацией. Можно выделить следующие варианты:

1. **Переменные окружения (Environment Variables):** Секреты передаются в среду запуска бинарника. Это самый универсальный и правильный подход с точки зрения 12-factor.
2. **Флаги при запуске:** Прокидывание значений через аргументы командной строки при запуске бинарника (менее безопасно, так как аргументы видны в списке процессов, например, через `ps`).
3. **Файлы `.env`:** Большинство систем и фреймворков поддерживают `.env` файлы, автоматически загружая их содержимое в переменные окружения (по сути, автоматизация пункта 1 на уровне самого инструмента).
4. **Специфичные конфигурационные файлы:** Некоторые инструменты ожидают наличие секретов в строго определенных местах своих конфигурационных файлов (YAML, JSON, TOML).
   - _4.1. С возможностью разделения:_ Инструменты, дающие возможность отделить секретный контент файла от несекретного (например, поддерживают интерполяцию переменных окружения вроде `${TOKEN}` внутри конфигурационного файла).
   - _4.2. Монолитные конфиги:_ Инструменты (например, Apprise), которые не поддерживают подстановку переменных и требуют, чтобы весь конфиг вместе с секретами лежал в одном файле.
5. **Специфика Docker и CI/CD (например, GitHub Actions):**
   - **Docker** имеет множество разрозненных способов работы с переменными: `ARG` (на этапе сборки), `secrets` (Docker Swarm/BuildKit), `env_file`, inline-переменные прямо внутри `docker-compose.yaml`.
   - **CI/CD (GitHub Actions):** Добавлять десятки переменных по отдельности в UI GitHub Secrets безумно неудобно. Из-за этого часто применяется костыльный паттерн: создается один секрет, содержащий весь конфиг (или всё содержимое `.env` файла) в виде одной длинной строки, которая затем парсится или записывается в файл при билде.

### 4. Проблема окружений, релизов и откатов

В кодовых проектах при строгом разделении конфигурации и реализации возникает проблема синхронизации релизов. Если новый код ожидает новую переменную окружения, а в конфигурации окружения (например, Production) её еще нет, деплой приведет к падению приложения. То же самое касается откатов: старый код может не уметь работать с новой структурой конфига.

**Решение:** Необходимо выстраивать процесс так, чтобы всегда сохранялась обратная совместимость. Конфигурацию и секреты нужно воспринимать как **внешний API** по отношению к приложению. Код должен быть написан так, чтобы уметь работать с отсутствующими (fallback значения) или устаревшими значениями во время переходных периодов деплоя.

### 5. Специфика Web-приложений (Client vs Server)

Отдельная сложность возникает в современных веб-приложениях (SPA, SSR), где есть клиентский код (выполняемый в браузере пользователя) и серверный код.

- Часть конфигурации (публичные API URL, ID систем аналитики) должна быть доступна на клиенте.
- Часть (приватные API ключи, доступы к БД) — строго только на сервере.
- **Момент инъекции:** Встраивать конфигурацию (даже публичную) на этапе сборки (**build-time**) — это антипаттерн, прямо нарушающий 12-factor (принцип "build once, deploy anywhere"). Вместо этого, при сборке статических бандлов (даже для SPA) должны оставаться плейсхолдеры. Реальные значения конфигурации должны инжектиться строго перед отдачей статики на веб-сервере (**run-time** / deploy-time), например, через `envsubst` (подмена переменных в `index.html` или сгенерированном `env.js`). Серверная конфигурация аналогично должна инжектиться только при запуске приложения.

## Существующие решения и подходы

Различные инструменты и утилиты по-разному решают описанные проблемы:

1. **Plaintext файлы (Локально):**
   - _Как работает:_ `.env` или `config.yaml` лежат локально и игнорируются в Git.
   - _Плюсы/Минусы:_ Простая настройка, но секреты хранятся на диске в открытом виде. Совершенно не масштабируется на команду разработчиков и усложняет синхронизацию.
2. **Шаблонизация (AOT - Ahead-of-Time) через Vault CLI (Doppler, 1Password, Infisical):**
   - _Как работает:_ В Git хранится безопасный шаблон (`config.template.yaml`). Специальный скрипт перед запуском или на этапе деплоя подтягивает секреты из облачного Vault и генерирует готовый файл конфигурации на диск.
   - _Плюсы/Минусы:_ Единственный адекватный способ работать с утилитами из пункта 4.2 (Apprise). Минус в том, что файл с секретами в итоге материализуется на диске.
3. **JIT (Just-In-Time) инъекция через обёртки (Doppler run / op run / openv):**
   - _Как работает:_ Процесс оборачивается в CLI-инструмент Vault'а. Секреты подтягиваются из сети и прокидываются в переменные окружения только в памяти процесса на время его работы.
   - _Плюсы/Минусы:_ Идеально решает проблемы пунктов 1 и 3 (максимальная безопасность). Однако плохо совместимо с системными фоновыми демонами (launchd/systemd), так как требует авторизации и сети прямо в момент старта сервиса.
4. **Нативная интеграция с Vault CLI:**
   - _Как работает:_ Инструмент сам поддерживает вызов внешних команд для получения кредов (например, `password-command: "op read..."` в resticprofile).
   - _Плюсы/Минусы:_ Безопасно, не требует внешних оберток, но поддерживается ничтожно малым количеством инструментов.
5. **Инструменты типа SOPS + sops-nix:**
   - _Как работает:_ Шифрует файлы с секретами прямо в Git. Интегрируется в процесс деплоя (например, через Nix) для расшифровки на лету.
   - _Плюсы/Минусы:_ Напрямую нарушает принцип 12-factor (секреты лежат вместе с кодом в репозитории). Усложняет ротацию ключей и онбординг новых людей по сравнению с полноценным SaaS Vault (Doppler/Infisical).

## Общие рекомендации по сценариям

### 1. Dotfiles (Локальное окружение)

В локальном окружении главная проблема — это интеграция с фоновыми процессами (например, `launchd` на macOS), которые запускаются вне контекста вашего терминала, а также утилиты со сложным конфигом (без поддержки env).

- **Рекомендация:** Использовать гибридный подход. Для сложных конфигов (Apprise) применять **AOT-шаблонизацию** (генерировать файлы при применении dotfiles). Для интерактивной работы в терминале и простых CLI-утилит использовать **JIT-инъекции / обертки** (например, `openv` для 1Password, чтобы переменные подтягивались автоматически при входе в директорию).

### 2. Self-hosted Home Lab (Docker Stack)

Множество сервисов, поднимаемых через `docker-compose`, которым нужны как переменные окружения, так и файлы-конфиги.

- **Рекомендация:** Использовать внешний централизованный менеджер секретов (Infisical, Doppler, 1Password). Для передачи секретов в контейнеры использовать автоматическую генерацию `.env` файлов (AOT) на хосте скриптом перед запуском `docker compose up`, либо интегрировать Vault-клиент прямо в скрипты запуска. Секретные файлы (сертификаты, сложные конфиги) прокидывать через Docker Secrets или биндить как read-only тома после их локальной генерации из шаблонов.

### 3. Dev Project (Веб-проект монорепа)

- **Рекомендация:**
  - **Единый источник истины:** Все секреты, фича-флаги и конфигурации для всех окружений (Dev/Staging/Prod) хранятся исключительно в специализированном Vault (Doppler, Infisical). Git не должен содержать `.env` файлов, кроме безопасных `*.example` или шаблонов.
  - **CI/CD Интеграция:** В GitHub Actions не нужно хранить копию конфигурации. Храните там только Service Token (один единственный токен) для доступа к Vault. На этапе пайплайна CI-раннер делает пулл нужных секретов из Vault для текущего собираемого окружения.
  - **Frontend (SPA):** Конфигурация (даже публичная) **не должна** встраиваться в статику на этапе сборки (build-time). CI-раннер собирает приложение один раз с конфигурационными плейсхолдерами. Во время развертывания (run-time / deploy-time), перед запуском Nginx/Caddy или внутри Node.js сервера, происходит инъекция переменных (например, подмена через `envsubst`), и только потом статика отдается клиентам.
  - **Backend:** На этапе сборки бекенда секреты не нужны. Доставка секретов происходит строго в **run-time** (на сервере деплоя). Это может быть обертка запуска (Doppler run) внутри Docker-контейнера или инъекция секретов из Vault напрямую в Kubernetes Secrets.
  - **Локальная разработка:** Разработчики авторизуются в Vault через CLI и запускают проект локально через JIT-обертки (например, `doppler run -- npm run dev`), получая всегда актуальную конфигурацию для локального окружения без необходимости вручную синхронизировать и плодить `.env` файлы.
  - **Отказоустойчивость:** Любое изменение конфигурации должно восприниматься как изменение API — код обязан иметь фоллбеки для плавного перехода.
