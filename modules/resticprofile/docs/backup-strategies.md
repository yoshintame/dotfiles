- Стратегии бэкапа: Local + B2

Сравнение трёх подходов к организации бэкапа в два хранилища (локальный диск + Backblaze B2).

**Текущая конфигурация:** используется стратегия 3 — два полностью независимых бэкап-процесса.

---

## Стратегия 1: Один restic-бэкап + rclone sync в B2

### Как работает

```
[файлы] → restic backup → [локальный repo] → rclone sync → [B2 bucket]
```

Restic делает бэкап в локальный репозиторий. После завершения — check, prune, forget.
Затем rclone копирует содержимое локального репозитория (как обычные файлы) в B2.

В resticprofile это реализуется через `run-after` хук:

```yaml
backup:
  run-after:
    - "rclone sync ~/Backups/restic/main b2:blackgaze/restic/main --transfers 8"
```

### Плюсы

- **Однократное сканирование файловой системы** — restic backup запускается один раз
- **Минимальная нагрузка на CPU** — данные шифруются только один раз (для локального repo), rclone копирует уже зашифрованные блоки как есть
- **Простая retention-политика** — prune/forget настраивается только для одного репозитория, B2 — точная зеркальная копия
- **Быстрое восстановление** — можно восстанавливать как из локального repo, так и напрямую из B2 (rclone-зеркало — это валидный restic-репозиторий)

### Минусы

- **Не нативный restic-метод** — rclone sync не знает о внутренней структуре restic-репозитория; синхронизация может произойти в момент, когда repo находится в промежуточном состоянии (между backup и prune)
- **Пропагация повреждений** — если локальный репозиторий повреждён, rclone добросовестно скопирует повреждения в B2
- **Невозможна независимая retention-политика** — B2 всегда является точной копией локального repo, нельзя хранить больше снепшотов в облаке
- **Зависимость от rclone** — дополнительный инструмент, который нужно устанавливать и поддерживать
- **Проблемы при конкурентном доступе** — если rclone sync запущен одновременно с другой операцией restic на том же repo, результат непредсказуем (смягчается lock-файлом в resticprofile)

## Стратегия 2: restic copy между двумя репозиториями

### Как работает

```
[файлы] → restic backup → [локальный repo] → restic copy → [B2 repo]
```

Restic делает бэкап в локальный репозиторий. Затем `restic copy` копирует снепшоты в отдельный B2-репозиторий. Это нативная команда restic для односторонней синхронизации.

Конфигурация resticprofile:

```yaml
development-local:
  inherit: repo-local
  backup:
    source:
      - "."
    source-base: "{{ .Env.HOME }}/Development"
  copy:
    initialize: true
    initialize-copy-chunker-params: true
    repository: "b2:blackgaze:restic/main"
    password-file: "{{ .Env.HOME }}/.config/resticprofile/development.key"
```

### Про re-encryption (перешифрование)

**Да, restic copy всегда делает re-encryption, даже если пароль одинаковый.**

Причина: пароль — это не ключ шифрования. Пароль используется для расшифровки мастер-ключа, который генерируется случайно при `restic init`. Два независимо инициализированных репозитория с одинаковым паролем имеют **разные мастер-ключи**. Поэтому restic copy обязан:

1. Расшифровать данные мастер-ключом исходного repo
2. Зашифровать мастер-ключом целевого repo

Это создаёт ощутимую CPU-нагрузку, особенно для больших объёмов данных.

### Важно: copy-chunker-params

При инициализации целевого репозитория **обязательно** использовать флаг `--copy-chunker-params` (в resticprofile: `initialize-copy-chunker-params: true`). Это гарантирует, что оба репозитория используют одинаковый алгоритм разбиения файлов на чанки. Без этого restic copy не сможет переиспользовать существующие блоки данных в целевом репозитории — каждый copy будет копировать всё заново.

### Плюсы

- **Нативный метод restic** — поддерживается самим restic, корректно работает с внутренней структурой репозитория
- **Однократное сканирование файловой системы** — backup запускается один раз, copy работает уже с данными в repo
- **Два независимых репозитория** — каждый самодостаточен, можно восстанавливать из любого
- **Независимая retention-политика** — можно настроить разные правила forget/prune для локального и B2 репозитория
- **Атомарность** — restic copy корректно обрабатывает снепшоты, нет риска скопировать репозиторий в промежуточном состоянии
- **Инкрементальность** — copy переносит только новые снепшоты и блобы, а не всё содержимое

### Минусы

- **Re-encryption** — CPU-нагрузка на перешифрование всех новых данных при каждом copy
- **Требуется доступ к обоим repo одновременно** — при copy restic держит подключение к обоим хранилищам
- **Более сложная конфигурация** — нужно правильно инициализировать целевой repo с copy-chunker-params
- **Зависимость от первого repo** — copy можно сделать только после успешного backup; если локальный repo недоступен, B2 не обновится

## Стратегия 3: Два независимых бэкап-процесса (текущая)

### Как работает

```
[файлы] → restic backup → [локальный repo]
[файлы] → restic backup → [B2 repo]
```

Два полностью независимых restic-профиля. Каждый делает свой backup, свой forget, свой prune. Репозитории никак не связаны.

### Плюсы

- **Максимальная независимость** — повреждение одного repo никак не влияет на другой
- **Простота конфигурации** — каждый профиль самодостаточен, нет зависимостей
- **Полностью независимая retention** — разные политики для локального и облачного хранения
- **Отказоустойчивость** — если B2 недоступен, локальный бэкап всё равно работает (и наоборот)
- **Параллельный запуск** — теоретически можно запускать оба бэкапа параллельно (разные repo, разные lock-файлы)

### Минусы

- **Двойное сканирование файловой системы** — restic backup дважды обходит одни и те же файлы (при N хранилищах — N раз)
- **Двойная CPU-нагрузка на шифрование** — данные шифруются независимо для каждого repo
- **Дублирование конфигурации** — source-списки, exclude-паттерны повторяются в каждом профиле (смягчается inheritance в resticprofile)
- **Рассинхронизация** — снепшоты в разных repo создаются в разное время, содержат разные состояния файлов
- **Удвоенное время бэкапа** — суммарное время = время локального бэкапа + время B2-бэкапа

## Disaster Recovery: восстановление на новом ноутбуке

**Сценарий:** ноутбук потерян/сломан. Все локальные файлы и локальный restic-репозиторий утрачены. Есть новый ноутбук, доступ к B2, пароль от репозитория (key-файл сохранён отдельно или запомнен).

**Цели:**

1. Восстановить рабочие файлы из B2
2. Воссоздать локальный restic-репозиторий для продолжения регулярных бэкапов

---

### Стратегия 1: rclone sync — восстановление

B2 содержит побайтовое зеркало локального repo. Это тот же самый репозиторий, просто лежит в облаке.

**Шаг 1: Восстановить файлы**

Можно работать с B2 напрямую как с restic-репозиторием (через rclone backend):

```bash
# Посмотреть доступные снепшоты
restic -r rclone:b2:blackgaze/restic/main snapshots

# Восстановить последний снепшот по тегу
restic -r rclone:b2:blackgaze/restic/main restore latest --tag home --target ~/
restic -r rclone:b2:blackgaze/restic/main restore latest --tag development --target ~/
```

**Шаг 2: Воссоздать локальный repo**

Просто скачать всё зеркало обратно:

```bash
mkdir -p ~/Backups/restic
rclone sync b2:blackgaze/restic/main ~/Backups/restic/main --transfers 8 --progress
```

После этого локальный repo — точная копия того, что было. Все снепшоты, вся история сохранена. Конфигурация resticprofile не требует изменений, всё продолжает работать как раньше.

**Оценка:**

- Восстановление файлов: простое, напрямую из B2
- Восстановление локального repo: **тривиальное** — rclone sync в обратном направлении
- Сохранение истории: **полная** — все снепшоты, тот же мастер-ключ, тот же repo
- Изменения в конфигурации: **не нужны**

---

### Стратегия 2: restic copy — восстановление

B2 содержит независимый restic-репозиторий со своим мастер-ключом. Все данные есть, но repo отличается от локального (другой ключ, возможно другой набор снепшотов из-за независимой retention).

**Шаг 1: Восстановить файлы**

```bash
# Посмотреть снепшоты в B2
restic -r b2:blackgaze:restic/main snapshots

# Восстановить
restic -r b2:blackgaze:restic/main restore latest --tag home --target ~/
restic -r b2:blackgaze:restic/main restore latest --tag development --target ~/
```

**Шаг 2: Воссоздать локальный repo**

Вариант A — **свежий старт** (просто, но без истории):

```bash
mkdir -p ~/Backups/restic
restic init --repo ~/Backups/restic/main --password-file ~/.config/resticprofile/development.key
# Далее обычный backup создаст первый снепшот
```

Локальный repo создаётся с нуля. История снепшотов в нём начинается заново. B2-repo при этом сохраняет всю старую историю.

Вариант B — **перенести историю из B2 в локальный** (сложнее, но сохраняет историю):

```bash
mkdir -p ~/Backups/restic

# Инициализировать локальный repo с chunker-params из B2
restic init --repo ~/Backups/restic/main \
  --password-file ~/.config/resticprofile/development.key \
  --copy-chunker-params --repo2 b2:blackgaze:restic/main \
  --password-file2 ~/.config/resticprofile/development.key

# Скопировать все снепшоты из B2 в локальный (обратный copy)
restic copy --repo b2:blackgaze:restic/main --repo2 ~/Backups/restic/main \
  --password-file ~/.config/resticprofile/development.key \
  --password-file2 ~/.config/resticprofile/development.key
```

Это re-encryption в обратном направлении: расшифровка B2 мастер-ключом, шифрование новым локальным мастер-ключом. Занимает время и CPU пропорционально объёму данных.

**Важный нюанс:** после восстановления по варианту B `copy` секция в resticprofile продолжает работать без изменений — B2 repo не изменился, а новый локальный repo просто получил копию его снепшотов. Однако при следующем `restic copy` local→B2 restic обнаружит, что снепшоты (скопированные обратно) уже есть в B2, и пропустит их.

**Оценка:**

- Восстановление файлов: простое, напрямую из B2
- Восстановление локального repo: **средняя сложность** — нужно выбрать между свежим стартом и полным copy
- Сохранение истории: **опционально** — вариант A теряет локальную историю, вариант B сохраняет (ценой времени на re-encryption)
- Изменения в конфигурации: **не нужны**

---

### Стратегия 3: независимые repo — восстановление

B2 содержит полностью независимый репозиторий. Он никак не связан с локальным.

**Шаг 1: Восстановить файлы**

```bash
restic -r b2:blackgaze:restic/main snapshots
restic -r b2:blackgaze:restic/main restore latest --tag home --target ~/
restic -r b2:blackgaze:restic/main restore latest --tag development --target ~/
```

**Шаг 2: Воссоздать локальный repo**

Локальный repo был полностью независимым, поэтому единственный вариант — создать новый:

```bash
mkdir -p ~/Backups/restic
restic init --repo ~/Backups/restic/main --password-file ~/.config/resticprofile/development.key
```

Первый `resticprofile backup` создаст первый снепшот. Локальная история начинается с нуля.

Технически можно сделать `restic copy` из B2 в новый локальный repo, но это **не предусмотрено** этой стратегией и потребует тех же шагов, что и вариант B из стратегии 2 (init с copy-chunker-params, затем copy). При этом в обычной работе по стратегии 3 chunker-params не синхронизируются, поэтому copy будет неэффективным — каждый блоб будет пересоздан.

**Оценка:**

- Восстановление файлов: простое, напрямую из B2
- Восстановление локального repo: **только с нуля** — старая локальная история потеряна безвозвратно
- Сохранение истории: **нет** для локального repo (история B2 не переносима из-за разных chunker-params)
- Изменения в конфигурации: **не нужны**

| Восстановление локального repo | Тривиально (rclone sync обратно) | Средне (init + обратный copy)      | Только с нуля           |

| Время восстановления repo      | Быстро (только скачивание)       | Долго (скачивание + re-encryption) | Мгновенно (пустой init) |

## Рекомендация

**Стратегия 2 (restic copy)** — оптимальный баланс между надёжностью и эффективностью:

- Файловая система сканируется один раз
- Оба репозитория полностью независимы и самодостаточны
- Нативный механизм restic, корректно работающий с внутренними структурами
- Можно настроить разную retention-политику для локального и облачного хранилища
- Единственный ощутимый минус — CPU на re-encryption, но для типичных объёмов домашних бэкапов это несущественно

## Практические советы

### Стоимость B2 и восстановление

Backblaze B2: порядка $0.005/GB/месяц за хранение, $0.01/GB за скачивание. Для disaster recovery важнее дёшево хранить долго; полная выкачка репозитория обходится дороже.

**Экономия при восстановлении из B2:** не скачивать весь репозиторий (rclone sync обратно), а восстанавливать нужное напрямую из B2 через `restic -r b2:... restore ...`. Restic подтянет только те блоки, которые нужны для выбранных снепшотов/путей — объём и стоимость скачивания меньше. Сначала восстанавливать из локального repo (если доступен), к B2 обращаться только при потере локального.

### Порядок операций (стратегия 1: rclone/rsync)

При автоматизации (systemd, launchd, cron): сначала полный цикл restic (backup → forget → prune при необходимости), затем синхронизация копии репозитория. Так удалённая копия не окажется в промежуточном состоянии. В resticprofile этого порядка добивается `run-after` после backup.

**Чем синхронизировать:** кроме rclone подойдёт **rsync** — побайтово копировать директорию repo на другой диск или по SSH. Идея та же: дождаться завершения restic, потом синхронизировать, чтобы все копии repo были в одном и том же состоянии ([r/restic](https://www.reddit.com/r/restic/comments/1o5mv55/create_copiesmirrors_of_a_restic_repository_in/)).

### Prune отдельно от backup

`restic prune` ресурсоёмкий. Имеет смысл запускать его реже, чем backup (например, раз в неделю или раз в месяц), по отдельному таймеру. Forget можно оставить после каждого backup; prune — по расписанию.

### Уведомления при сбоях

При автоматическом запуске бэкапов стоит настроить оповещение при падении (например, systemd `OnFailure=...` с отправкой статуса юнита по почте или в мессенджер). Иначе сбой можно не заметить до момента восстановления.

### Холодное хранилище (Glacier, Deep Archive)

Если вторая копия репозитория уезжает в «холодное» хранилище (AWS Glacier, S3 Deep Archive и т.п.): проверка целостности (`restic check`) по такой копии приводит к запросам на извлечение данных и **высоким расходам**. Имеет смысл не гонять check по cold storage без необходимости. При синхронизации через rclone в такой бакет рекомендуют включать **`--immutable`** (append-only), чтобы случайно не перезаписать или не удалить объекты.

## Несколько устройств / источников: один repo или отдельный на устройство?

**Сценарий:** бэкапится несколько устройств или независимых источников (ноутбук + NAS, несколько машин) в одно и то же хранилище (локальный диск или B2). Вопрос: один общий репозиторий на всё или отдельный репозиторий на каждое устройство?

**Рекомендация из сообщества restic** ([issue #1015](https://github.com/restic/restic/issues/1015), [forum](https://forum.restic.net/t/best-practice-for-multiple-hosts-to-single-repository/1687)):

- **Один общий репозиторий на каждое место назначения** — предпочтительно. Restic дедуплицирует данные по всему репозиторию: одинаковые файлы и чанки с разных устройств хранятся один раз. Для облака (B2) это даёт экономию места и стоимости.
- **Организация не страдает:** у каждого снепшота сохраняется hostname клиента, по нему видно, с какого устройства восстановлять. Дополнительно можно использовать теги или отдельные пароли на устройство для разграничения доступа.
- **Несколько хостов могут писать в один repo одновременно** — restic корректно обрабатывает параллельную запись (lock).

Отдельный repo на устройство имеет смысл, если нужна жёсткая изоляция (разные политики retention, разные владельцы данных) или если источники логически не связаны («мои данные» vs «общий NAS семьи»).
