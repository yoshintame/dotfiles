# Стратегии бэкапа: Local + B2

Сравнение трёх подходов к организации бэкапа в два хранилища (локальный диск + Backblaze B2).

**Текущая конфигурация:** используется стратегия 2 — restic copy (backup в локальный repo, затем copy в B2).

---

## Стратегия 1: Один restic-бэкап + rclone sync в B2

```
[файлы] → restic backup → [локальный repo] → rclone sync → [B2 bucket]
```

Restic делает бэкап в локальный репозиторий. После завершения rclone копирует содержимое репозитория в B2.

**Плюсы:**

- Однократное сканирование ФС
- Минимальная нагрузка на CPU — данные шифруются один раз
- Простая retention — B2 = точная зеркальная копия
- Восстановление: B2-зеркало — вал**идный restic-репозиторий

**Минусы:**

- Не нативный restic-метод — rclone не знает о структуре restic-repo
- Пропагация повреждений — повреждённый repo скопируется в B2
- Невозможна независимая retention
- Зависимость от rclone

---

## Стратегия 2: restic copy (текущая)

```
[файлы] → restic backup → [локальный repo] → restic copy → [B2 repo]
```

Restic делает бэкап в локальный репозиторий. Затем `restic copy` копирует снепшоты в отдельный B2-репозиторий.

### Про re-encryption

restic copy **всегда** делает re-encryption, даже если пароль одинаковый. Два независимо инициализированных репозитория имеют разные мастер-ключи. При copy данные расшифровываются мастер-ключом источника и шифруются мастер-ключом цели.

### copy-chunker-params

При инициализации целевого репозитория обязательно `--copy-chunker-params` (в resticprofile: `initialize-copy-chunker-params: true`). Гарантирует одинаковую разбивку на чанки — без этого каждый copy будет копировать все данные заново.

**Плюсы:**

- Нативный метод restic — корректная работа с внутренней структурой
- Однократное сканирование ФС
- Два полностью независимых репозитория
- Независимая retention для каждого repo
- Атомарность — нет риска промежуточных состояний
- Инкрементальность — переносятся только новые снепшоты

**Минусы:**

- Re-encryption — CPU-нагрузка при каждом copy
- Требуется доступ к обоим repo одновременно
- Нужна правильная инициализация с copy-chunker-params

## Стратегия 3: Два независимых бэкап-процесса

```
[файлы] → restic backup → [локальный repo]
[файлы] → restic backup → [B2 repo]
```

Два полностью независимых профиля. Каждый делает свой backup, forget, prune.

**Плюсы:**

- Максимальная независимость — повреждение одного repo не влияет на другой
- Простота конфигурации
- Отказоустойчивость — если B2 недоступен, локальный backup работает

**Минусы:**

- Двойное сканирование ФС (N хранилищ = N раз)
- Двойная CPU-нагрузка на шифрование
- Рассинхронизация — снепшоты в разное время

## Сравнительная таблица


| Критерий                    | 1. rclone sync      | 2. restic copy (текущая) | 3. Независимые |
| -----------------------------| ---------------------| --------------------------| ----------------|
| Сканирование ФС             | 1 раз               | 1 раз                    | N раз          |
| Шифрование                  | 1 раз               | 2 раза (re-encrypt)      | N раз          |
| Нативный метод restic       | Нет                 | Да                       | Да             |
| Независимость репозиториев  | Нет (зеркало)       | Да                       | Да             |
| Независимая retention       | Нет                 | Да                       | Да             |
| Устойчивость к повреждениям | Низкая (пропагация) | Высокая                  | Максимальная   |
| Консистентность снепшотов   | Зависит от rclone   | Высокая                  | Нет            |
| Суммарная нагрузка на CPU   | Минимальная         | Средняя                  | Максимальная   |

## Disaster Recovery: сравнение


| Критерий                       | 1. rclone sync                   | 2. restic copy (текущая)           | 3. Независимые   |
| --------------------------------| ----------------------------------| ------------------------------------| ------------------|
| Восстановление файлов          | Просто                           | Просто                             | Просто           |
| Восстановление локального repo | Тривиально (rclone sync обратно) | Средне (init + обратный copy)      | Только с нуля    |
| Сохранение локальной истории   | Полное                           | Опционально (с re-encryption)      | Невозможно       |
| Время восстановления repo      | Быстро (только скачивание)       | Долго (скачивание + re-encryption) | Мгновенно (init) |

## Практические советы

### Стоимость B2

Backblaze B2: ~$0.005/GB/мес за хранение, ~$0.01/GB за скачивание. Для disaster recovery — восстанавливать нужное напрямую из B2 через `restic restore`, а не скачивать весь repo.

### Prune отдельно от backup

`restic prune` ресурсоёмкий. Запускается еженедельно (по воскресеньям), а не после каждого backup.
